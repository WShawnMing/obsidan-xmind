# Obsidian 思维导图插件需求细化

## 1. 项目背景

用户希望在 Obsidian 中直接使用类似 XMind、飞书思维导图的脑图能力，而不是在外部工具整理完再贴回笔记。插件除了常规思维导图编辑外，还需要支持从 Markdown 文档的层级结构自动生成思维导图。

这意味着产品本质上不是单独的脑图编辑器，而是一个以 Obsidian 笔记为中心、围绕 Markdown 结构化内容展开的思维导图插件。

## 2. 产品目标

## 2.1 核心目标

1. 在 Obsidian 内打开或生成思维导图视图，不跳出当前知识库工作流。
2. 支持从 Markdown 的层级结构自动生成思维导图。
3. 提供接近 XMind / 飞书思维导图的基础操作体验。
4. 支持在节点标题中直接使用和点击 `[[wiki link]]`，把脑图节点作为笔记入口。
5. 保证 Markdown 与脑图之间无感切换，不生成独立内容中间产物。
6. 保证笔记结构和脑图结构之间关系清晰、可追踪、可维护。

## 2.2 非目标

1. 第一阶段不追求完整替代 XMind 的高级图形能力。
2. 第一阶段不做多人实时协同。
3. 第一阶段不做复杂的演示模式、甘特图、鱼骨图、组织架构图等多图形类型。
4. 第一阶段不做跨文件自动聚合成一个大脑图。
5. 第一阶段不生成独立 `.xmind`、`.json`、`.mm` 之类的内容文件作为脑图主存储。

## 3. 目标用户

1. 用 Obsidian 做读书笔记、项目规划、会议纪要的个人用户。
2. 希望把已有 Markdown 结构快速可视化的用户。
3. 喜欢先列提纲，再用脑图整理结构的用户。

## 4. 参考产品拆解

## 4.1 借鉴 XMind 的部分

1. 以节点为中心的编辑方式。
2. 快捷键驱动的新建同级 / 子级节点。
3. 节点折叠、展开、拖拽调整顺序。
4. 主题和布局切换能力。

## 4.2 借鉴飞书思维导图的部分

1. 轻量、快速、偏文档式的结构编辑体验。
2. 与文本层级天然对应的操作方式。
3. 降低用户从“文本提纲”切换到“脑图结构”的成本。

## 5. 用户场景

## 5.1 场景 A：从 Markdown 快速生成脑图

用户已有一篇结构清晰的 Markdown 文档，希望一键生成脑图，用来梳理大纲和层次。

成功标准：

1. 用户能从当前笔记直接打开脑图视图。
2. 标题、列表层级能稳定转成节点层级。
3. 结构结果基本符合用户预期，不需要大量手工修正。

## 5.2 场景 B：边写提纲边看脑图

用户在 Markdown 中写提纲，同时希望实时或手动刷新脑图视图，以便从视觉上检查结构是否合理。

成功标准：

1. Markdown 变更后，脑图能刷新。
2. 刷新行为可预期，不会丢失当前视图状态。

## 5.3 场景 C：在脑图中做结构调整

用户在脑图中增删改节点、调整层级和顺序，希望这些操作最终能回写到对应笔记。

成功标准：

1. 节点编辑简单直接。
2. 回写后 Markdown 结构不混乱。
3. 遇到无法安全回写的情况时有明确提示。

## 5.4 场景 D：把节点作为笔记入口

用户希望脑图中的每个节点都能直接关联 Markdown 笔记，尤其是标题中存在 `[[Note]]`、`[[Note|Alias]]` 这类 wiki link 时，能够在脑图中直接点击跳转。

成功标准：

1. 节点标题中的 `[[...]]` 能正确识别和显示。
2. 点击链接可打开对应笔记。
3. 未解析成功的链接有清晰的未解析态，而不是静默失效。

## 5.5 场景 E：Markdown 与脑图无感切换

用户不希望“导入 Markdown 生成脑图”或“导出脑图生成 Markdown”成为两个显式步骤，而是希望同一篇笔记既能以文本看，也能以脑图看。

成功标准：

1. 切换视图不生成额外内容文件。
2. 内容修改始终回到同一篇 Markdown。
3. 只有布局编排相关信息作为额外元数据持久化。

## 6. 功能分层

## 6.1 MVP 必做

1. 新增 Obsidian 思维导图视图。
2. 从当前 Markdown 笔记生成思维导图。
3. 支持标题层级解析。
4. 支持列表层级解析。
5. 支持节点展开 / 折叠。
6. 支持缩放、拖拽平移、节点选中。
7. 支持新增子节点、同级节点。
8. 支持节点重命名、删除。
9. 支持节点顺序调整。
10. 支持将节点结构回写为 Markdown。
11. 支持识别、渲染、点击节点标题内的 `[[wiki link]]`。
12. 不生成独立脑图内容文件，Markdown 与脑图在同一内容源上切换。
13. 支持基础 layout 元数据持久化，例如节点位置偏移、画布视口。
14. 支持基础主题样式。

## 6.2 第二阶段建议

1. 支持双向联动高亮。
2. 支持节点备注、标签、优先级、任务状态。
3. 支持局部刷新，不必整图重建。
4. 支持多种布局。
5. 支持节点手动尺寸调整和更自由的排版能力。
6. 支持导出 PNG / SVG / Markdown。
7. 支持节点快捷菜单和更完整快捷键。

## 6.3 后续增强

1. 支持附件、外链、嵌入块等更丰富的节点内容。
2. 支持跨笔记引用节点。
3. 支持模板和主题系统。
4. 支持演示模式。
5. 支持与 Canvas 互操作。

## 7. 关键产品决策

## 7.1 核心对象是什么

建议将“当前 Markdown 笔记”作为主数据源。也就是说，脑图首先是某篇笔记的可视化视图，而不是独立文件格式。

这样做的好处：

1. 与 Obsidian 工作流一致。
2. `Markdown -> 脑图` 需求天然成立。
3. 数据存储简单，不容易形成两份真相。
4. 可以做到 Markdown 和脑图无感切换。

代价：

1. 脑图编辑必须考虑如何安全回写 Markdown。
2. 某些纯图形化操作会受 Markdown 结构约束。

## 7.2 是否允许生成独立脑图内容文件

建议不允许。也就是说，插件不生成独立 `.xmind` 或脑图 JSON 作为主内容文件。

允许的额外持久化内容只有：

1. layout 元数据
2. 视图状态
3. 插件设置

这样做的好处：

1. 没有“导入 / 导出”心智负担。
2. 不会出现 Markdown 和脑图内容各改各的双份冲突。
3. 更符合你提出的“无感转换”要求。

## 7.3 是否首发就做双向编辑

建议做“有限双向”而不是“完整自由编辑”。

建议首发支持：

1. 改节点标题。
2. 增删子节点 / 同级节点。
3. 调整同一父节点下的顺序。
4. 调整节点父子关系，但仅在能明确映射为 Markdown 层级时允许。

建议暂缓：

1. 任意拖拽成复杂自由布局。
2. 与 Markdown 结构不一致的浮动节点。

## 7.4 内容编辑与布局编辑是否区分

建议明确区分，而且这是核心交互原则。

1. 内容编辑：改标题、增删节点、改父子关系、改顺序，会回写 Markdown。
2. 布局编辑：拖拽位置、调整视觉大小、分支展开状态，只写 layout 元数据，不改 Markdown 内容。

建议默认优先把鼠标拖拽理解为布局编辑，避免用户一次拖动就误改正文结构。

## 7.5 wiki link 是否作为一等公民

建议是。

也就是说，节点标题中的 `[[...]]` 不只是普通文本，而是：

1. 解析时要识别
2. 渲染时要可点击
3. 编辑时要原样保留
4. 回写时要无损输出

## 7.6 Markdown 解析优先级

建议优先支持以下两类结构：

1. 标题树：`#` 到 `######`
2. 列表树：`-`、`*`、`1.` 等嵌套列表

并定义统一规则：

1. 文档主标题作为根节点。
2. 标题节点下面的列表，挂在该标题节点之下。
3. 列表缩进代表子节点。
4. 普通段落作为节点备注，不直接变成子节点。
5. 节点标题中的 `[[wiki link]]` 保留为节点标题内容的一部分。

## 8. Markdown -> 思维导图规则

## 8.1 根节点规则

按优先级选择根节点：

1. 第一条 `# 一级标题`
2. 文件名
3. 默认文案“Untitled Mind Map”

## 8.2 标题规则

1. `#` 表示一级节点，`##` 表示其子节点，依此类推。
2. 当标题层级跳跃时，自动挂到最近的上级标题节点下。
3. 标题下的正文内容默认归为该节点的备注。

示例：

```md
# 项目规划
## 背景
## 目标
### 本周目标
```

对应结构：

```text
项目规划
├── 背景
└── 目标
    └── 本周目标
```

## 8.3 列表规则

1. 同级列表项为同级节点。
2. 缩进列表项为子节点。
3. 有序列表和无序列表在脑图中统一表示为普通节点。
4. 任务列表 `- [ ]` / `- [x]` 在第二阶段映射为节点状态。

示例：

```md
# 读书计划
- 第一章
  - 概念
  - 案例
- 第二章
```

对应结构：

```text
读书计划
├── 第一章
│   ├── 概念
│   └── 案例
└── 第二章
```

## 8.4 标题与列表混合规则

建议按“标题建骨架，列表补分支”的方式处理。

示例：

```md
# 会议纪要
## 风险
- 排期延迟
- 资源不足
## 行动项
- Alice 跟进方案
- Bob 完成排查
```

对应结构：

```text
会议纪要
├── 风险
│   ├── 排期延迟
│   └── 资源不足
└── 行动项
    ├── Alice 跟进方案
    └── Bob 完成排查
```

## 8.5 暂不纳入结构节点的内容

MVP 中以下内容不直接转成树节点：

1. 普通段落
2. 引用块
3. 代码块
4. 表格
5. 图片

这些内容可作为节点备注保留，或在首版先忽略并提示“部分内容未映射”。

## 8.6 wiki link 规则

1. 节点标题中允许包含 0 到多个 `[[wiki link]]`。
2. `[[Note]]`、`[[Note|Alias]]` 都要保留并识别。
3. 脑图渲染时，链接部分应有可点击态。
4. 点击链接默认打开对应笔记。
5. 未解析的链接应显示为未解析态，但仍保留原文。
6. 回写 Markdown 时，`[[...]]` 语法必须原样保留，不应被转成纯文本。

## 8.7 无感转换规则

1. 不存在“导入 Markdown”这个独立动作，打开脑图视图就是解析当前笔记。
2. 不存在“导出回 Markdown”这个独立动作，内容编辑天然回到当前笔记。
3. 脑图和 Markdown 的内容层必须是一份数据。
4. 唯一允许的额外产物是 layout 相关元数据。
5. layout 继承采用保守策略，无法稳定匹配时直接删除旧 layout 并回退自动布局。

## 9. 交互设计建议

## 9.1 视图入口

1. 命令面板：`Open mind map for current note`
2. 右键菜单：在当前文件打开脑图
3. 侧边栏图标：快速打开脑图视图

## 9.2 基础交互

1. 单击选中节点
2. 双击或回车编辑节点文本
3. `Tab` 新增子节点
4. `Enter` 新增同级节点
5. `Delete / Backspace` 删除节点
6. 鼠标滚轮缩放
7. 空白区域拖拽平移
8. 点击折叠按钮收起 / 展开分支
9. 点击节点标题中的 `[[wiki link]]` 打开对应笔记
10. 拖拽节点默认调整 layout，而不是直接修改 Markdown 结构

## 9.3 编辑保护

以下情况需要提示或限制：

1. 当前文件不是 Markdown
2. 文件内容结构异常，无法稳定解析
3. 用户脑图操作无法安全映射回 Markdown
4. 文件被外部改动导致回写冲突

## 10. 验收标准

## 10.1 MVP 验收

1. 打开任意结构化 Markdown 笔记，可成功生成脑图。
2. 至少支持 500 个节点内的流畅浏览与基础编辑。
3. 常规标题 / 列表混合文档解析正确率达到可用水平。
4. 节点增删改和排序后，可稳定回写 Markdown。
5. 节点标题中的 `[[wiki link]]` 可稳定识别、显示和跳转。
6. 插件不生成独立脑图内容文件。
7. 插件重启后，视图状态、layout 元数据和配置能恢复。
8. 当旧 layout 无法与最新 Markdown 稳定对应时，系统会自动删除失效 layout，而不是错误复用。

## 10.2 易用性验收

1. 新用户在 1 分钟内能从笔记打开脑图。
2. 常用快捷键符合常见脑图工具直觉。
3. 解析失败时有明确错误说明，而不是静默失败。

## 11. 推荐开发节奏

## Phase 1

1. 搭建 Obsidian 插件骨架
2. 只读脑图视图
3. Markdown 解析器
4. wiki link 识别与跳转
5. 树布局与基础交互

## Phase 2

1. 节点编辑
2. Markdown 回写
3. 视图状态与 layout 元数据管理
4. 基础主题

## Phase 3

1. 双向联动
2. 更自由布局与尺寸调整
3. 导出
4. 节点元信息
5. 更完整快捷键与菜单

## 12. 当前推荐结论

如果要把这个模糊需求快速收敛成“第一版能做什么”，建议定义为：

“一个 Obsidian 插件，为当前 Markdown 笔记提供思维导图视图，支持依据标题与列表层级生成脑图，并支持对节点进行基础结构编辑后回写 Markdown。”

结合你新增的要求，现在更准确的定义应该是：

“一个以当前 Markdown 笔记为唯一内容源的 Obsidian 思维导图插件，支持依据标题、列表和标题内的 `[[wiki link]]` 生成脑图；内容编辑直接回写同一份 Markdown，只有布局编排相关信息作为额外元数据持久化。”

这个定义已经足够具体，可以直接进入原型和工程实现。
